💡 TIP: The savant-workspace MCP server connects your AI sessions (Copilot, Claude, Codex) directly to SAVANT. Any tool below can be called from your terminal — changes show up here instantly with a toast notification.

🧭 Workspace Management

🧭
get_current_workspace
Auto-detect which workspace this AI session belongs to. Returns workspace name, description, priority, status, and task summary counts.
📋
list_workspaces
List all workspaces. Filter by status: open, closed, or all.
🏗️
create_workspace
Create a new workspace with name, description, and priority (critical/high/medium/low).
📌
assign_session_to_workspace
Assign the current AI session to a workspace by name or ID. Auto-detects session.
🔒
close_workspace
Close a workspace. Defaults to your current workspace if no ID is given.
🔍
get_workspace
Get a workspace by ID or fuzzy name match. Returns full details with task counts.

☑ Task Management

📋
list_tasks
Get tasks for a workspace. Defaults to current workspace and today's date. Filter by status: todo, in-progress, done, blocked, or all.
➕
create_task
Create a task and assign it to a workspace. Provide title, description, priority, and status.
✏️
update_task
Update fields on an existing task — title, description, status, priority.
✅
complete_task
Mark a task as done. Syncs to the dashboard instantly.
🎯
get_next_task
Get the highest-priority actionable task (todo or in-progress). Priority order: critical → high → medium → low.
🔗
add_task_dependency
Add a dependency link: task_id depends on depends_on. Both tasks must exist. Circular dependencies are rejected.
✂️
remove_task_dependency
Remove a dependency link between two tasks.

📝 Session Notes

📖
list_session_notes
Get all notes for a session. Defaults to the current session. Notes render as Markdown.
🗒️
create_session_note
Create a note on a session. Supports Markdown. Visible in the SAVANT dashboard under the session's Notes tab.
🗑️
delete_session_note
Delete a note by index (0-based, oldest first). Defaults to the current session.

✏️
update_merge_request
Update fields on an existing merge request — title, status, priority, author, jira link.
📋
list_merge_requests

🔍
get_merge_request

📌
assign_mr_to_session

🔓
unassign_mr_from_session
Remove a merge request assignment from a session.
💬
add_mr_note
Add a comment/note to a merge request. Attributed to the current session.
📖
list_mr_notes
Get all notes/comments for a merge request.

📋
list_jira_tickets

list_rules

list_policies

💡 TIP: The savant-context MCP server provides semantic code search and memory bank access across all your indexed repositories. AI agents use this to find relevant code, documentation, and project context without manual file browsing.

🔍 Semantic Search

🔎
code_search
Semantic code search across indexed repos. Accepts a natural-language query q, optional repo filter (name or list), limit (1–100, default 10), and exclude_memory_bank flag. Returns file paths, language, line range, content snippets, and distance scores.
🧠
memory_bank_search
Semantic search within memory bank markdown files only. Same params as code_search but restricted to memory bank documents (files in memory/ directories).

📚 Memory Bank Resources

📋
memory_resources_list
List memory bank resources from the database. Optional repo filter (name or list). Returns file paths, repo names, and chunk counts for each memory document.
📖
memory_resources_read
Read a memory bank resource by URI (file path). Returns the full content of the memory bank document for AI context injection.

📁 Repository Management

📂
repos_list
List all indexed repos with README excerpts. Optional filter for substring name matching. Returns repo name, path, file/chunk counts, status, and last indexed timestamp.
📊
repo_status
List per-repo index status counts — files indexed, chunks stored, indexing status (added/indexing/indexed/error), and last indexed time.

⚡ Quick Start Workflow

// 1. Search for authentication code across all repos code_search(q: "JWT token validation middleware") // 2. Search only in a specific project code_search(q: "database connection pool", repo: "savant-app") // 3. Find relevant memory bank docs memory_bank_search(q: "deployment architecture") // 4. List all memory bank resources memory_resources_list() // 5. Read a specific memory doc memory_resources_read(uri: "memory/architecture.md") // 6. Check which repos are indexed repo_status()
🗄️ How it works: Projects are indexed by walking the file tree (respecting .gitignore), chunking files into ~500-line segments with 50-line overlap, generating 768-dim embeddings with stsb-distilbert-base, and storing vectors in sqlite-vec for sub-millisecond KNN search. The embedding model auto-downloads (~260MB) on first use.
📁 Memory Bank: Files inside a memory/ directory at the project root are automatically tagged as memory bank resources. These are typically architecture docs, decisions, and context files that AI agents can reference for high-level understanding.

🗂️ Node Taxonomy

client

🗂️
domain

⚙️
service

📦
library
Shared gems & packages — icn-user-acl, icn-entitlements, icn-jwt-gem, info-share-gem
🔧
technology
Infrastructure & frameworks — Ruby on Rails, React, PostgreSQL, Redis, Elasticsearch, Kubernetes, Sidekiq, Kong
💡
insight
Curated developer knowledge — architecture decisions, lessons learned, integration notes, debugging findings
📁
project
Project- or workspace-level nodes that group related work, repos, and insights
🏷️
concept
Abstract ideas and patterns — RBAC, PKCE, ownership models, rollout strategies
📦
repo
Repository/codebase nodes that connect services, sessions, and implementation knowledge
💬
session
AI coding session nodes used to preserve execution context, handoffs, and work history
🐛
issue
Known bugs, blockers, and debt items that should stay connected to the rest of the graph

🔍 Search & Retrieve

🔎
search
Search nodes by text. query (required), node_type filter (client | domain | service | library | technology | insight | project | concept | repo | session | issue), limit (default 20). Returns matching nodes with IDs, types, and content.
🕐
recent
Get the most recently added knowledge nodes. Optional node_type and limit (default 20).
🌐
neighbors
Traverse the graph from a node. node_id (required), depth 1–5 (default 1), optional edge_type filter. Returns connected nodes & edges up to N hops away.
🗺️
project_context
Aggregate full context for a workspace. Traverses the graph (depth 2) from the workspace project node to return connected insights, services, domains, tasks, and notes. Use at session start to onboard an agent. Requires workspace_id.
📋
list_concepts
List all technology nodes (Rails, Redis, Kubernetes…). Returns node_id and title — useful for finding IDs to wire connect() calls.

✏️ Write & Connect

💾
store
Store a curated insight. content (required), source (session | task | note), workspace_id, repo, files (comma-separated), connections (JSON array of {node_id, edge_type} pairs to auto-link on creation).
🔗
connect
Create a typed edge between two nodes. source_id, target_id, edge_type (relates_to | learned_from | applies_to | uses | evolved_from | contributed_to | part_of | integrates_with | depends_on | built_with), optional label.
✂️
disconnect
Remove an edge between two nodes. Provide source_id + target_id. Optional edge_type to remove only a specific edge; omit to remove all edges between the pair.

⚡ Quick Start Examples

// Store an architecture decision linked to the icn service node store(content: "Migrated auth to OKTA PKCE flow — removed legacy session cookies", source: "session", workspace_id: "ws_123", connections: '[{"node_id":"kgn_icn","edge_type":"applies_to"}]') // Search for everything about authentication search(query: "authentication OKTA", node_type: "insight") // Explore what Fidelity integrates with (depth 2) neighbors(node_id: "kgn_client_fidelity", depth: 2, edge_type: "integrates_with") // Get full context before starting work on a workspace project_context(workspace_id: "ws_123")
